# 1. Что это за продукт

Мы делаем не generic page builder и не low-code CMS.
Мы делаем специализированный визуальный редактор HTML-презентаций.

Цель:
- открыть существующий HTML презентации;
- показать максимально правдивое live preview;
- позволить выбирать и редактировать допустимые элементы;
- менять текст, изображения, GIF, видео, базовые стили, часть структуры;
- вставлять новые элементы и layout presets;
- экспортировать обратно чистый HTML без editor-мусора.

Ключевое ощущение:
«Открыл → выбрал → изменил → сохранил».

# 2. Неподвижные инварианты

Нельзя нарушать:
- no dead ends;
- predictable UX;
- truthful preview;
- recoverability через undo/redo/autosave;
- Basic/Advanced;
- никаких служебных editor-артефактов в export;
- shell-UI живёт снаружи контента, а не загрязняет документ;
- архитектуру iframe + bridge + modelDoc ломать нельзя.

# 3. Зафиксированная архитектура

Основа:
- Parent shell
- Iframe preview
- Bridge
- modelDoc

Parent shell отвечает за:
- topbar
- slides list
- inspector
- floating toolbar
- context menu
- insert palette / insert entry point
- history
- autosave
- export
- mobile shell
- diagnostics UI

Iframe отвечает за:
- живой запуск документа
- реальный DOM презентации
- выполнение презентационных скриптов
- runtime selection/editing через bridge

Bridge отвечает за:
- команды parent -> iframe
- сообщения iframe -> parent
- runtime metadata
- selection
- element/slide updates
- diagnostics / heartbeat

modelDoc отвечает за:
- каноническую модель документа
- export
- restore
- history source
- editor logic

# 5. Что сейчас считается source of truth

Держись этих принципов:
- редактор должен быть понятен слабому пользователю;
- типовые действия должны делаться за 1–2 шага;
- shell не должен уезжать, ломаться или перекрывать canvas;
- topbar должен оставаться чистым и глобальным;
- локальные действия должны жить в inspector / floating toolbar / context menu;
- insert UX должен быть контекстным и не жить постоянным шумом над слайдом;
- темы и responsive должны быть системными, а не набором хаотичных overrides;
- direct manipulation должен быть удобным, но не ломать DOM и не конфликтовать с deck logic;
- export обязан оставаться чистым;
- любые новые доработки сначала проверяй на адаптив, тему, UX-смысл и отсутствие архитектурной деградации.

# 8. Как писать код

Это критично.

- Не плодить повторные top-level функции.
- Не создавать новые “historical override layers”.
- Не делать хаотичные CSS patches в конце файла.
- Если правишь shell/layout/theme, думай как о системе, а не как о точечном костыле.
- Любая UI-команда должна иметь один источник правды.
- Состояния должны быть явными, а не размазаны по скрытым флагам.
- Если добавляешь новый UI-state, фиксируй его осознанно.
- Если рефакторишь — делай аккуратно, не ломая архитектуру и текущий working path.

# 10. Что важно не забыть

- Если правишь UI, следи за консистентностью, адаптивностью и отсутствием визуальных артефактов.
- Не расширяй продукт хаотично.
- Если есть конфликт между новой фичей и устойчивостью — выбирай устойчивость.
- Если есть конфликт между мощью и понятностью — выбирай понятность, но без потери архитектуры.
- Действуй итеративно, пока оставшиеся проблемы не будут закрыты и код не станет чище, структурированнее, а UX — устойчиво рабочим.

Начни работу с:
1. краткого восстановления контекста,
2. перечисления 3–5 главных проблем,
3. описания следующего конкретного инженерного прохода,
4. затем переходи к правкам.
