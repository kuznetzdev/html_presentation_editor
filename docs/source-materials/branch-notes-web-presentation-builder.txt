## 1. Что это за проект

Мы делаем **универсальный визуальный редактор HTML-презентаций**. Его задача — не просто показать слайды, а дать возможность редактировать уже готовую HTML-презентацию без ручной правки кода: выбирать элементы, менять текст, изображения, стили, структуру и потом экспортировать чистый HTML обратно. Это зафиксировано в уточнённом ТЗ: редактор должен работать не только с текстом, но и с любыми элементами, поддерживать изображения, вставку новых блоков, базовое оформление и синхронизацию изменений в модель документа. fileciteturn1file0 fileciteturn1file2

При этом у проекта есть важное осознанное ограничение: редактор **не гарантирует сохранение сложной анимации, переходов и кастомной интерактивной логики**, если пользователь ломает критичную DOM-структуру, классы или обёртки, от которых зависит поведение презентации. Это не баг, а часть продуктовой рамки: мы редактируем контент и базовое оформление, а не обещаем полную совместимость со всем пользовательским JS. fileciteturn1file0 fileciteturn1file4

## 2. Что именно мы хотим получить в итоге

Целевое состояние проекта такое:

- загружается **любой разумный HTML deck**: reveal.js, generic HTML, кастомная вёрстка;
- внутри `iframe` превью работает **максимально близко к обычному открытию HTML в браузере**;
- в режиме редактирования можно выбирать **любой элемент слайда**, а не только текст;
- правки происходят через удобный UI: инспектор, контекстное меню, плавающую панель, горячие клавиши;
- все изменения уходят в **модель документа** и потом в корректный экспорт;
- служебные атрибуты редактора не попадают в итоговый HTML. fileciteturn1file1 fileciteturn1file2

То есть проект — это уже не “демо-редактор текста”, а **мини-приложение для визуального монтажа HTML-презентаций**.

## 4. Почему именно такая архитектура

Проект упёрся в то, что реальные HTML-презентации часто живут не как “статические `div.slide`”, а как полноценные мини-приложения. На реальном тестовом deck’е `prepodovai_v3_edit_with_embedded_screens.html` видно, что:
- слайды абсолютно позиционируются;
- видимость контролируется через `.slide.active`;
- есть собственные стили, анимации и логика навигации. fileciteturn1file19

Из этого следует ключевой вывод: **нельзя строить редактор так, чтобы он грубо переписывал DOM превью, массово скрывал слайды или вырезал рабочие части документа**. Поэтому правильная схема — держать:
- **живое превью** внутри `iframe`,
- **модель документа** в основном окне,
- **bridge** между ними,
- и синхронизировать только осмысленные изменения.

С точки зрения браузерного стека это верный фундамент:
- для обмена между родительским окном и `iframe` штатно используется `postMessage`;
- для отслеживания DOM-изменений штатно используется `MutationObserver`;
- для reveal.js корректнее опираться на его API `getSlides()`, `getCurrentSlide()`, `slide()` вместо грубого `display:none`. citeturn286516view5turn286516view6

## 7. Где сейчас основные риски

Вот здесь как раз твои замечания надо считать не комментарием, а **официальным списком техрисков проекта**.

### 7.1. Гонки между parent и iframe
Сейчас логика сложная:
- bridge внутри `iframe` меняет DOM;
- parent параллельно обновляет `modelDoc`;
- сверху появляются `history`, `autosave`, контекстные действия, дублирование, вставка картинок.

Это означает риск гонок:
- быстрые клики;
- undo сразу после `element-updated`;
- autosave на частично синхронизированном состоянии;
- context menu action во время незавершённой синхронизации.

Это **главный технический риск №1**.

### 7.2. Рассинхрон modelDoc и реального iframe DOM
Ты правильно отметил: операции вроде `insert-element`, `move-element`, `duplicate-element` фактически живут в двух местах:
- сначала действие применяется в `iframe`,
- потом модель патчится через `element-update` / `slide-update`.

Это правильный подход по идее, но он чувствителен к:
- порядку children;
- потере сообщений;
- ошибкам при переиспользовании `nodeId`;
- обновлению слайда целиком вместо точечного узла.

Это **главный риск №2**.

### 7.3. Чувствительность к чужим CSS и layout
Resize/drag/selection geometry завязаны на реальную геометрию внутри `iframe`, а она зависит от:
- `position`,
- `transform`,
- `overflow`,
- flex/grid-контекста,
- абсолютного позиционирования,
- собственных анимаций deck’а.

Значит любые drag/resize механики будут очень чувствительны к чужим стилям.  
Это **главный риск №3**.

### 7.4. Потенциальный конфликт со скриптами презентации
Некоторые презентации могут сами перехватывать:
- `click`,
- `contextmenu`,
- `keydown`,
- pointer events.

Поэтому режим редактирования потенциально должен уметь:
- временно подавлять часть скриптов презентации;
- либо переводить deck в “soft read-only interaction mode”.

Это **главный риск №4**.

## 8. Что делаем дальше

Дальше проект надо вести **в двух параллельных потоках**.

### Поток A. Стабилизация ядра
Сначала добиваем надёжность архитектуры.

#### A1. Единый контракт синхронизации
Нужно жёстко зафиксировать:
- кто главный для состояния в момент изменения;
- какие команды идут parent → iframe;
- какие события идут iframe → parent;
- где считается “операция завершена”.

Практически: для каждой операции нужен lifecycle:
`command -> apply in iframe -> ack/update -> patch model -> history snapshot -> autosave`

#### A2. Нормальная история изменений
Undo/Redo уже нужен, но он не должен быть просто “слепок HTML на каждый чих”.  
Нужен хотя бы гибрид:
- структурные операции — snapshot;
- частые style/input изменения — debounce/throttle;
- группировка мелких изменений в одну команду.

#### A3. Периодическая сверка modelDoc
Твоё предложение правильное: нужен **периодический reconciliation loop**, который раз в N секунд сверяет модель и состояние активных слайдов/узлов. Это не замена event-driven sync, а страховка от рассинхрона.

#### A4. MutationObserver внутри bridge
Это уже прописано в ТЗ как правильная архитектурная опора: новые runtime-элементы должны автоматически получать `data-editor-node-id`. fileciteturn1file1  
Это надо довести до production-like поведения.

### Поток B. UX-слой “под человека”
Это уже второй уровень — сделать редактор не просто рабочим, а удобным.

## 9. Приоритеты по UX на следующую итерацию

### Высокий приоритет
1. **Умное контекстное меню**  
   Секционное, с иконками, разное для текста / изображения / контейнера, с Escape и безопасным позиционированием.

2. **Плавающая панель нового уровня**  
   Контекстная, перетаскиваемая, со сворачиванием, с разным набором кнопок по типу элемента.

3. **Сворачиваемые секции инспектора**  
   С сохранением состояния в `localStorage`.

4. **Toasts вместо “глухих” действий**  
   “Элемент удалён”, “Стили сброшены”, “Изображение заменено”, “Сохранено”.

5. **Копировать стиль / вставить стиль**  
   Это дешёвая по реализации, но очень сильная UX-фича.

### Средний приоритет
6. **Палитра элементов**  
   Текст / картинка / видео / фигура / вставить HTML.

8. **Ctrl+F по элементам**  
   Поиск по тексту / id / class / tag.

9. **Автоподсказка для больших изображений**  
   “Картинка слишком большая — вписать по ширине слайда?”

## 10. Что считаем “готовым” на ближайший этап

Ближайшая цель — не “идеальный редактор”, а **стабильный универсальный alpha build**.

### Минимально готово, когда:
- можно загрузить reveal.js deck и generic HTML deck;
- можно выбрать любой элемент;
- можно удалить / дублировать / заменить изображение / вставить текст и картинку;
- можно менять базовые стили и размеры;
