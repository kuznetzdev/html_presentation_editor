# SOURCE OF TRUTH  
## Проект: HTML Presentation Editor

**Версия:** 1.0  
**Статус:** основной документ проекта  
**Назначение:** единая истина по продукту, UX, архитектуре, ограничениям и критериям готовности

---

# 1. Что мы строим

Мы строим **локальный/клиентский визуальный редактор HTML-презентаций**, который позволяет:

- загрузить существующий HTML-файл презентации;
- показать превью максимально близко к тому, как HTML открывается в браузере;
- выбрать любой допустимый элемент слайда;
- изменить текст, изображения, видео, базовые стили и структуру;
- вставлять новые элементы;
- экспортировать назад **чистый HTML** без следов редактора.

Это **не** generic page builder, **не** сайт-конструктор и **не** low-code система общего назначения.  
Это **специализированный редактор HTML-презентаций**.

---

# 2. Главная цель продукта

Сделать редактор, который:

- не требует обучения;
- не допускает тупиков;
- понятен даже слабому пользователю;
- не ломает презентацию при обычных действиях;
- даёт ощущение качественного современного продукта;
- работает стабильно как инструмент редактирования, а не как технодемка.

Главное пользовательское ощущение:

**«Открыл → выбрал → изменил → сохранил»**

---

# 3. Неподвижные инварианты проекта

Это правила, которые нельзя нарушать.

## 3.1. Нет тупиков
Каждый экран обязан:
- иметь минимум одно явное действие;
- иметь понятный способ вернуться назад;
- объяснять текущее состояние;
- при отсутствии данных показывать следующий шаг, а не пустоту.

## 3.2. Всегда есть primary action
На каждом ключевом экране есть одно главное действие.

Примеры:
- Открыть
- Продолжить
- Сохранить
- Экспортировать
- Вставить

## 3.3. Простота по умолчанию
Сложность скрыта.  
Пользователь не должен видеть внутреннюю механику редактора, если она ему не нужна.

## 3.4. Контекстность
Показываем только те инструменты, которые нужны для текущего элемента и текущей задачи.

## 3.5. Предсказуемость
Редактор не делает неожиданных структурных изменений и не перестраивает интерфейс хаотично.

## 3.6. Recoverability
Практически любое значимое действие обратимо:
- undo;
- redo;
- автосохранение;
- восстановление.

## 3.7. Превью — источник доверия
Превью должно быть максимально честным.  
Пользователь должен верить тому, что видит в холсте.

## 3.8. Редактор не загрязняет результат
В экспортируемом HTML не должно оставаться:
- `data-editor-*`;
- bridge script;
- helper styles;
- runtime selection state;
- служебных editor-артефактов.

---

# 4. Что продукт должен уметь обязательно

## 4.1. Базовый обязательный контур
Редактор обязан уметь:

- открыть HTML из файла;
- открыть HTML из вставленного текста;
- показывать список слайдов;
- рендерить live preview внутри iframe;
- переключаться между Preview и Edit;
- выбирать элементы;
- редактировать текст;
- заменять изображения;
- вставлять текст;
- вставлять изображение / GIF;
- вставлять видео;
- вставлять простой блок;
- вставлять layout из двух колонок;
- удалять элемент;
- дублировать элемент;
- менять базовые стили;
- экспортировать чистый HTML;
- сохранять черновик;
- восстанавливать черновик;
- делать undo/redo.

## 4.2. Поддерживаемые типы сущностей
Редактор работает как минимум с:

- text node / text block
- image
- GIF
- video
- iframe video embed
- block/container
- slide
- simple layout preset

---

# 5. Что продукт не должен делать

Это важно зафиксировать жёстко.

Редактор **не обязан**:

- гарантировать полную совместимость со всеми кастомными JS-скриптами презентации;
- сохранять любую сложную анимацию при глубоком structural editing;
- быть полноценным CSS-конструктором уровня Webflow;
- быть универсальным редактором любых сайтов;
- обещать полную безошибочную round-trip-совместимость со всеми нестандартными deck-frameworks.

Если пользователь ломает критическую DOM-структуру, редактор работает в режиме **best effort**, а не магического абсолютного сохранения всего поведения.

# 6. Архитектура проекта — жёстко зафиксированная

Архитектура уже выбрана. Её не надо ломать. fileciteturn2file1

## 6.1. Основа
Редактор строится на схеме:

- **Parent shell** — внешний интерфейс редактора
- **Iframe preview** — живое превью презентации
- **Bridge** — канал связи parent ↔ iframe
- **modelDoc** — каноническая модель документа

## 6.2. Роли компонентов

### Parent shell отвечает за:
- topbar;
- список слайдов;
- inspector;
- floating toolbar shell;
- context menu shell;
- history;
- autosave;
- export;
- mobile shell;
- diagnostics UI.

### Iframe отвечает за:
- живой запуск документа;
- реальный DOM презентации;
- работу презентационных скриптов;
- selection/runtime-операции через bridge.

### Bridge отвечает за:
- выбор элементов;
- сообщения о runtime state;
- применение точечных изменений;
- sync активного DOM в модель;
- heartbeat и runtime diagnostics.

### modelDoc отвечает за:
- хранение текущей редактируемой версии документа;
- экспорт;
- откат;
- восстановление состояния;
- канонический DOM для editor logic.

# 7. Архитектурные запреты

## 7.1. Нельзя превращать editor в DOM-грязь
Нельзя массово:
- оборачивать презентацию editor-wrapper’ами;
- вставлять в export служебные узлы;
- ломать исходную структуру без необходимости.

## 7.2. Нельзя делать parent главным редактором реального DOM
Редактирование живого документа идёт через bridge.  
Parent не должен хаотично лезть в DOM iframe как в свою песочницу.

## 7.3. Нельзя выстраивать продукт вокруг “костыльных перезагрузок всего”
Полная пересборка iframe допустима как fallback, но не как основной путь каждой операции.

## 7.4. Нельзя тянуть редактор в сторону generic page builder
Любая новая функция проверяется вопросом:
**она делает HTML presentation editor лучше или тащит нас в конструктор сайтов?**

# 8. Режимы работы

## 8.1. Preview mode
Preview mode нужен для:
- честного просмотра;
- проверки внешнего вида;
- навигации по слайдам;
- проверки поведения deck.

В этом режиме:
- selection UI скрыт;
- editor overlay не мешает;
- редактирование отключено;
- интерфейс не вмешивается в DOM презентации без нужды.

## 8.2. Edit mode
Edit mode нужен для:
- выбора элементов;
- редактирования контента;
- изменения базовых стилей;
- вставки элементов;
- структурных операций.

В этом режиме:
- активны inspector, toolbar, context menu;
- доступна вставка;
- доступны destructive actions;
- включаются защитные механизмы.

# 9. UX-модель редактора

Пользователь должен воспринимать продукт как:

**«PowerPoint/Canva-подобный редактор, но для живого HTML»**

Не как IDE.  
Не как HTML-редактор с визуальным фасадом.  
Не как CMS.  
Не как builder с сотнями абстракций.

# 10. Basic / Advanced — обязательная модель сложности

## 10.1. Basic
Это дефолтный режим продукта.

Basic показывает только:
- тип элемента;
- содержимое;
- базовые стили;
- быстрые действия;
- вставку базовых сущностей.

Basic должен закрывать большую часть типовых задач.

## 10.2. Advanced
Advanced раскрывает:
- позиционирование;
- размеры;
- margin/padding;
- фон/границы;
- id/class/data-*;
- transform;
- HTML элемента/слайда;
- дополнительные структурные действия.

## 10.3. Правило
Если фича нужна только опытному пользователю и не нужна для массового сценария — она живёт в Advanced.

# 12. Topbar — source of truth

## 12.1. Что должно быть видно
Слева:
- иконка редактора;
- название;
- имя текущего документа / краткий статус.

Справа:
- Preview / Edit;
- Открыть;
- Экспорт;
- Настройки;
- Справка.

## 12.2. Что не должно висеть в topbar постоянно
Не держим на виду:
- engine diagnostics;
- технические runtime-метки;
- лишние debug-индикаторы;
- громоздкие dirty-state блоки;
- всё, что не нужно обычному пользователю каждую секунду.

## 12.3. Настройки
В Settings:
- тема;
- Basic / Advanced;
- подсказки;
- diagnostics;
- reset UI-state;
- при необходимости developer tools.

Она открывается по запросу.

# 20. Mobile / responsive — source of truth

Не просто “сжиматься”, а оставаться usable.

## 20.2. При ширине <1024

## 20.3. Mobile command rail

## 20.4. Touch-инварианты

# 21. Синхронизация parent ↔ iframe ↔ modelDoc — source of truth

Это самый важный технический узел.

## 21.1. Любая операция проходит цикл
**command → apply in iframe → ack/update → patch modelDoc → history → autosave**

## 21.2. Seq / revision обязательны
Каждое изменение получает ревизию.

## 21.3. lastAppliedSeq
Shell применяет sync только если ревизия новее.

## 21.4. Slide lock
После shell-driven изменения входящий sync по затронутому слайду временно блокируется на короткое окно.

## 21.5. Debounce / throttle обязательны
Особенно для:
- input;
- частых style updates;
- burst mutation scenarios.

## 21.6. Reconciliation loop обязателен
Нужен периодический reconciliation между `modelDoc` и live runtime.

---

# 22. MutationObserver — source of truth

## 22.1. Наблюдаем не всё подряд
Нельзя вешать тяжёлый observer на весь document без контроля.

## 22.2. Наблюдаем контейнер слайдов
Либо ближайший общий container для runtime slides.

## 22.3. Следим только за нужным
Следить за:
- class;
- style;
- hidden;
- aria-hidden;
- aria-current;
- src/media-critical changes.

## 22.4. Не триггерим sync на editor noise
Изменения `data-editor-*` не должны считаться meaningful runtime changes.

# 23. Критические элементы — source of truth

## 23.1. Что считаем критическими
Например:
- `.reveal`
- `.slides`
- `.shower`
- `.deck`
- `.deck-container`
- `.remark-slide-container`

## 23.2. Что с ними делать
Либо:
- не выбирать совсем;
- либо выбирать только для просмотра информации.

## 23.3. Что запрещено
Для них нельзя:
- delete;
- dangerous resize;
- dangerous position changes;
- unsafe HTML replacement;
- structural wrapping.

# 24. История изменений — source of truth

## 24.1. Undo/Redo обязательны
Для:
- текста;
- стилей;
- вставки;
- удаления;
- дублирования;
- HTML replace;
- media operations.

## 24.2. MVP history
Допустим snapshot + debounce.

## 24.3. Целевая history
Command-based / diff-based system.

## 24.4. UX-правило
Undo/Redo не должны пугать пользователя полной ломкой превью.

---

# 25. Автосохранение — source of truth

## 25.1. Что хранить
- HTML;
- активный слайд;
- режим;
- тема;
- Basic/Advanced;
- inspector collapse state;
- toolbar position;
- UI-state.

## 25.2. Когда сохранять
- после серии изменений;
- с debounce;
- безопасно и незаметно.

## 25.3. Восстановление
При запуске:
- показываем banner;
- источник;
- дату;
- восстановить;
- удалить черновик.

# 26. Экспорт — source of truth

## 26.1. Основной обязательный экспорт
Чистый HTML.

## 26.2. Перед экспортом
- дотянуть sync;
- проверить актуальность модели;
- зачистить editor artifacts.

## 26.3. Удаляем из export
- `data-editor-*`
- bridge
- helper styles
- contenteditable
- spellcheck
- runtime markers
- preview-only base

## 26.4. Проверяем уникальность id
Дубликаты устраняются.

## 26.5. Критерий
Экспортированный HTML открывается без редактора и не выглядит как “проколотый editor-логикой”.

## 29.1. Потеря bridge
Если bridge молчит слишком долго:
- показать banner;
- заблокировать unsafe editing;
- предложить reload preview.

## 29.2. Runtime errors iframe
Не должны убивать shell.  
Показываем тост, логируем, держим редактор живым.

## 29.3. Все опасные операции через try/catch
Никаких silent failures там, где пользователь ждёт результат.

# 32. Definition of Done

Редактор считается готовым к следующему серьёзному уровню только если:

## UX
- интерфейс не перегружен;
- типовые задачи решаются в Basic;
- нет тупиков;
- toolbar/menu/palette не мешают canvas.

## Функционально
- можно открыть, выбрать, исправить, вставить, удалить, дублировать, экспортировать.

## Технически
- нет типовых рассинхронов;
- history работает;
- autosave работает;
- recovery работает;
- bridge-loss не рушит продукт.

## Адаптивно
- редактор usable на desktop и mobile-width shell.

## Экспорт
- HTML чистый;
- editor-мусора нет;
- результат открывается отдельно.
