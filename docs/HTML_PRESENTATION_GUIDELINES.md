# HTML Presentation Guidelines for the Editor

Этот документ задаёт практический контракт для разработчиков и AI-агентов,
которые создают HTML-презентации под `HTML Presentation Editor`.

Цель не в том, чтобы сделать "красивый HTML вообще", а в том, чтобы сделать
deck, который редактор честно открывает, предсказуемо выделяет, безопасно
редактирует и чисто экспортирует.

Главный принцип:

`Open -> select -> edit -> save`

Если deck ломает этот путь, deck сделан плохо для этого продукта, даже если
он визуально эффектный.

## 1. Что редактор считает качественной презентацией

Качественная презентация для этого редактора:

- состоит из реальных slide-root DOM-узлов, а не из виртуальных экранов,
  собираемых JavaScript после загрузки
- имеет стабильную и предсказуемую структуру, которую можно выделять по
  элементам
- не прячет полезный контент за системными обёртками, transform-хаками и
  невидимыми overlay-слоями
- использует нормальный HTML и CSS, а не canvas-only или shadow-DOM-first
  подход
- сохраняет редактируемый текст и медиа как реальные DOM-узлы
- не требует знания внутренностей редактора, но уважает его импортный
  контракт

## 2. Рекомендуемый канонический контракт

Используйте именно этот паттерн, если хотите максимальную совместимость.

```html
<!doctype html>
<html lang="ru">
<head>
  <meta charset="UTF-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />
  <title>Deck title</title>
  <style>
    :root {
      --bg: #f4f4f6;
      --surface: #ffffff;
      --text: #101828;
      --muted: #667085;
      --accent: #2563eb;
    }

    * { box-sizing: border-box; }
    body {
      margin: 0;
      background: var(--bg);
      font-family: Inter, Arial, sans-serif;
    }

    .deck {
      display: grid;
      gap: 24px;
      padding: 24px;
    }

    .slide {
      width: 1280px;
      height: 720px;
      position: relative;
      overflow: hidden;
      background: var(--surface);
      color: var(--text);
      border-radius: 24px;
      box-shadow: 0 24px 60px rgba(16, 24, 40, 0.12);
    }

    .title {
      position: absolute;
      left: 72px;
      top: 72px;
      width: 840px;
      margin: 0;
      font-size: 64px;
      line-height: 1.02;
    }

    .body {
      position: absolute;
      left: 72px;
      top: 190px;
      width: 620px;
      margin: 0;
      font-size: 28px;
      line-height: 1.45;
      color: var(--muted);
    }
  </style>
</head>
<body>
  <main class="deck" data-doc-type="presentation" data-version="1.0">
    <section class="slide" data-slide-id="intro">
      <h1
        class="title slide-title"
        data-node-id="intro-title"
        data-node-type="text"
        data-editable="true"
      >
        Заголовок слайда
      </h1>
      <p
        class="body"
        data-node-id="intro-body"
        data-node-type="text"
        data-editable="true"
      >
        Короткий поддерживающий текст.
      </p>
    </section>
  </main>
</body>
</html>
```

## 3. Обязательные правила структуры

### 3.1 Слайды

Предпочтительный формат:

- один slide = один реальный DOM-узел
- используйте `<section class="slide" data-slide-id="...">`
- держите все слайды одновременно в DOM
- используйте уникальный `data-slide-id` для каждого слайда

Редактор умеет распознавать и другие структуры (`section.slide`,
`div.slide`, `.deck .slide`, Reveal/Shower-подобные контейнеры), но это
fallback, а не лучший путь.

Не рассчитывайте на эвристику, если вы контролируете разметку.

### 3.2 Элементы внутри слайда

Для значимых узлов используйте:

- `data-node-id` для стабильной идентичности
- `data-node-type` для явного типа узла
- `data-editable="true"` только там, где текст или содержимое реально должны
  редактироваться

Поддерживаемые authored-типы:

- `text`
- `group` или `container`
- `image` или `img`
- `video` или `iframe`
- `table`
- `cell` или `table-cell`
- `code` или `code-block`
- `svg`
- `fragment` или `stateful`
- `shape` или `element`

Если authored-тип не указан, редактор будет угадывать его по DOM. Это хуже,
чем задать тип явно.

### 3.3 Заголовок слайда

Чтобы редактору было проще строить summary и navigation, у слайда должен быть
явный title-сигнал:

- `h1`, `h2` или `h3` внутри slide-root
- или `.slide-title`
- или `[data-slide-title]`

Лучший вариант: иметь нормальный heading и при желании дублировать смысл через
`data-slide-title`.

## 4. Что делать, чтобы deck хорошо редактировался

### Делайте

- держите slide-root в `position: relative`
- держите редактируемые блоки как реальные DOM-элементы
- используйте предсказуемые контейнеры: semantic blocks, grid, flex,
  абсолютное позиционирование без transform-трюков
- выносите повторяющееся оформление в CSS-классы и CSS variables
- добавляйте authored-маркеры к ключевым узлам, а не ко всему подряд
- делайте текст текстом, а не рендером в canvas, SVG path или background-image
- делайте изображения через `<img>` там, где картинка должна заменяться из UI
- держите DOM достаточно плоским, чтобы выделение элемента было очевидным

### Не делайте

- не стройте слайды через runtime-генерацию после загрузки
- не прячьте контент внутри shadow DOM
- не делайте весь слайд одним гигантским overlay-контейнером поверх полезных
  узлов
- не используйте `transform` для базовой компоновки, если можно обойтись grid,
  flex, `top/left`, `margin`, `padding`
- не делайте ключевой текст картинкой
- не делайте layout, который зависит от таймеров, поздней гидрации или внешних
  API
- не добавляйте `data-editor-*` вручную: это внутренний контракт редактора
- не полагайтесь на случайные `id` вместо `data-slide-id` и `data-node-id`

## 5. Чего редактор не любит

Следующие паттерны делают редактирование хуже или блокируют безопасные
операции:

- `transform` на slide-root
- `transform` на родителях редактируемых блоков
- скрытые элементы, перекрывающие живой контент
- глубокие вложенные absolute-контейнеры без authored-маркеров
- тяжёлые stateful-виджеты, которые постоянно переписывают свой DOM
- full-slide interactive wrappers, которые забирают pointer events
- нестабильные layout-паттерны, где координаты зависят от JS-вычислений

Продукт умеет честно блокировать часть direct manipulation в таких сценариях,
но хороший deck должен не провоцировать эти блокировки без необходимости.

## 6. Layout-стратегия, которая даёт лучший UX

Для этого редактора лучшие презентации обычно попадают в один из трёх паттернов:

- semantic layout: `section > header/main/footer`, `grid`, `flex`,
  переиспользуемые классы
- presentation layout: slide-root + 4-12 позиционируемых блоков с явными
  authored-id и authored-type
- hybrid layout: semantic каркас плюс 1-3 абсолютных декоративных слоя

Худший паттерн:

- десятки декоративных div-обёрток без типа и роли
- визуал построен через `transform: translate(...) scale(...)`
- текстовые блоки перекрываются прозрачными контейнерами

## 7. Asset policy

Для предсказуемого импорта и export parity:

- предпочитайте локальные относительные пути к ассетам
- держите рядом папку ассетов, если deck не self-contained
- не завязывайтесь на внешние CDN без крайней необходимости
- не используйте remote fonts как единственный источник типографики
- для изображений, которые пользователь должен менять, используйте `<img>`
- если в CSS есть `url(...)`, держите пути стабильными и относительными

Редактор умеет резолвить `src`, `href`, `poster`, `srcset`, inline `style` и
`<style>`-блоки, но чем проще путь к активу, тем надёжнее результат.

## 8. Semantic contract для AI-генерации

Если deck генерирует AI, просите не "сделать красивую HTML-презентацию", а
"сделать HTML deck под конкретный editor contract".

AI должен знать:

- слайды должны быть реальными DOM-узлами на момент загрузки
- предпочитаемый slide-root: `<section class="slide" data-slide-id="...">`
- каждый важный блок должен иметь `data-node-id`
- каждый важный блок должен иметь `data-node-type`
- редактируемый текст должен иметь `data-editable="true"`
- layout должен быть чистым, без лишних transform-зависимостей
- deck должен быть пригоден для safe visual editing, а не только для просмотра

## 9. Шаблон промпта для AI-агента

Используйте этот шаблон почти дословно.

```text
Сделай готовую HTML-презентацию для HTML Presentation Editor.

Контекст продукта:
- это visual editor для existing HTML slide decks
- ключевой путь пользователя: Open -> select -> edit -> save
- презентация должна не просто красиво выглядеть, а хорошо импортироваться,
  выделяться по элементам, безопасно редактироваться и чисто экспортироваться

Жёсткий контракт:
- верни один полный HTML-файл
- все слайды должны существовать в DOM сразу после загрузки
- каждый слайд делай как <section class="slide" data-slide-id="...">
- оборачивай все слайды в один контейнер .deck или <main class="deck">
- размер каждого слайда: 1280x720
- slide-root должен быть position: relative и overflow: hidden
- каждому важному редактируемому или значимому блоку дай уникальный
  data-node-id
- каждому важному блоку дай data-node-type из списка:
  text, container, image, video, table, table-cell, code-block, svg, fragment,
  element
- для редактируемых текстовых блоков ставь data-editable="true"
- не используй data-editor-* атрибуты
- не генерируй слайды через JavaScript после загрузки
- не используй shadow DOM
- не делай layout, который зависит от transform для базовой компоновки
- не делай ключевой текст картинками
- избегай невидимых overlay-слоёв поверх контента
- CSS делай аккуратным и переиспользуемым: классы, CSS variables, без хаоса
- если есть изображения, используй обычные <img> и относительные пути

Качество структуры:
- у каждого слайда должен быть явный заголовок через h1/h2/h3 или .slide-title
- DOM должен быть понятным человеку
- редактируемые элементы должны легко находиться и не быть buried в 6 уровнях
  вложенности
- лучше меньше, но чище DOM-узлов

Выход:
- верни только готовый HTML-файл без пояснений
- код должен быть production-ready и readable
```

## 10. Чеклист перед handoff

Перед тем как считать deck готовым, проверь:

- у каждого слайда есть `data-slide-id`
- все слайды существуют в DOM без runtime-сборки
- у важных блоков есть `data-node-id`
- authored `data-node-type` не выдуман, а взят из поддерживаемого словаря
- редактируемый текст отмечен `data-editable="true"`
- нет `data-editor-*`
- нет критического layout на `transform`
- нет прозрачных перекрывающих контейнеров поверх контента
- ключевой текст остаётся текстом
- структура слайда понятна без чтения кастомного JS
- ассеты локальны, относительны и не завязаны на хрупкие внешние URL

## 11. Краткое правило принятия решений

Если вы сомневаетесь между:

- более эффектной HTML-конструкцией
- более простой и честной структурой для визуального редактирования

выбирайте второе.

Для этого продукта предсказуемость структуры важнее, чем декоративная
изощрённость DOM.
