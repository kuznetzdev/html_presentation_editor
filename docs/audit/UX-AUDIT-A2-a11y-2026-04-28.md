## Static findings

### F1 — CRIT — WCAG 4.1.2 (A) — Modal dialogs missing dialog semantics
**File:** `editor/presentation-editor.html:1667, 1764, 1802, 1868`
**Pain:** Four modals (`#openHtmlModal`, `#htmlEditorModal`, `#videoInsertModal`, `#shortcutsModal`) are plain `<div class="modal">` with only `aria-hidden="true"`. Screen readers do not announce them as dialogs, do not switch reading mode, and do not announce the heading on open. Only `#layerPicker:1932` has `role="dialog"`.
**WCAG ref:** 4.1.2 Name, Role, Value (Level A) + 2.4.6 Headings & Labels.
**Evidence:** `<div class="modal" id="openHtmlModal" aria-hidden="true">` (no role, no aria-modal, no aria-labelledby).
**Fix direction:** S — add `role="dialog" aria-modal="true" aria-labelledby="<modal>Title"` to all four modal containers + give each `<h3>` an id. Minimal HTML edit (12 attrs total).
**Reuse note:** v2.0.22 focus-ring + v2.0.16 contrast bumps don't address structural dialog semantics. Independent fix.

### F2 — CRIT — WCAG 2.1.1 (A) — Context menu items not reachable via Tab
**File:** `editor/src/context-menu.js:497-505` (rendering); `editor/src/shell-overlays.js:677-707` (keydown handler)
**Pain:** Context-menu buttons are created with `button.tabIndex = -1` (line 505) and **no `role="menuitem"`** on the buttons or `role="menu"` properly wired with `aria-activedescendant`. The keydown handler attaches to the menu container (`els.contextMenu.addEventListener("keydown")`), so navigation works only when the menu has DOM focus. After `closeTransientShellUi`, focus is `body` — Arrow keys do nothing. The menu wrapper (`#contextMenu`) has `role="menu"` (HTML:1925) but its children are `<button>` without `role="menuitem"` — axe will warn `aria-required-children` on next loaded-edit scan.
**WCAG ref:** 2.1.1 Keyboard (A); 4.1.2 Name/Role/Value (A).
**Evidence:** `button.tabIndex = -1; ... button.innerHTML = '<span class="menu-icon">...'` — no role attribute set.
**Fix direction:** S — add `button.setAttribute("role", "menuitem")` in renderContextMenu loop; ensure `focusContextMenuButton(0)` runs after `is-open` (already does on `:787`, good); add `aria-orientation="vertical"` on `#contextMenu`.
**Reuse note:** Tabindex pattern matches palette/topbar overflow which work — only the role is missing.

### F3 — HIGH — WCAG 1.4.3 (AA) — Experimental "Beta" badge fails AA contrast
**File:** `editor/styles/base.css:153-168`
**Pain:** `.experimental-badge { color: rgb(138, 95, 0); background: rgba(255, 175, 38, 0.18); font-size: 10px; }` — orange-on-light-orange small text. Estimated ratio after alpha-blend onto `#ffffff`: foreground ~`#8a5f00` on effective `#fff1d8` ≈ 4.0:1 — fails AA (needs 4.5:1). Worse on `--shell-panel-soft` (#f9f9fb) where the alpha tint shifts. Used on `#exportPptxBtn` (Beta).
**WCAG ref:** 1.4.3 Contrast (Minimum) — Level AA, normal text 10px.
**Evidence:** `font-size: 10px` + `rgba(255, 175, 38, 0.18)` background.
**Fix direction:** S — bump font-size to 11px (matches `--text-xs`) and darken color to `#6b4a00` (≥ 4.5:1 on the alpha-blended background). Or migrate background to opaque `#fff3d6` for predictability.
**Reuse note:** v2.0.16 muted-text bump targeted `--shell-text-muted` only, doesn't touch this hard-coded literal.

### F4 — HIGH — WCAG 1.3.1 (A) — Slide-template menu, insert-palette, topbar overflow lack `aria-orientation` and items lack `role="menuitem"` consistency
**File:** `editor/presentation-editor.html:272-297` (slideTemplateBar), `:428-502` (quickPalette), `:188-233` (topbarOverflowMenu)
**Pain:** All three containers use `role="menu"` correctly. But:
- `quickPalette` items have `role="menuitem"` ✓
- `slideTemplateBar` items have `role="menuitem"` ✓
- `topbarOverflowMenu` items have `role="menuitem"` ✓
- However NONE declares `aria-orientation` (default is "vertical" but JAWS+NVDA behave better when explicit; insert-palette is actually horizontal in some layouts).
**WCAG ref:** 1.3.1 Info & Relationships (A) + ARIA Authoring Practices.
**Evidence:** `<div id="topbarOverflowMenu" class="topbar-overflow-menu" role="menu" aria-hidden="true" aria-label="Дополнительные команды topbar">` — no aria-orientation.
**Fix direction:** S — add `aria-orientation="vertical"` to `#topbarOverflowMenu`, `#contextMenu`, `#slideTemplateBar`; add `aria-orientation="horizontal"` to `#quickPalette`.
**Reuse note:** Independent of focus-ring / contrast work.

### F5 — HIGH — WCAG 1.3.1 (A) — `#previewFrame` iframe `title="Preview"` is English in a Russian UI
**File:** `editor/presentation-editor.html:558`
**Pain:** Whole UI is `<html lang="ru">` with Cyrillic copy. The iframe announces as "Preview" in screen readers — inconsistent for Russian users using NVDA-RU/JAWS-RU. Per `keyboard-nav.spec.js:538-583` (test 6), Cyrillic invariant is enforced for slide-item aria-labels and slide-action titles, but NOT for the iframe title.
**WCAG ref:** 1.3.1 Info & Relationships (A) + 3.1.2 Language of Parts (AA).
**Evidence:** `<iframe id="previewFrame" class="hidden" title="Preview" allowfullscreen></iframe>`
**Fix direction:** S — change to `title="Превью презентации"`. One-line edit.
**Reuse note:** Not covered by v2.0.x foundation; a Cyrillic-string regression is already enforced for slide-rail (`keyboard-nav.spec.js`) — extend the same pattern.

### F6 — HIGH — WCAG 2.4.1 (A) — No skip link to main workspace
**File:** `editor/presentation-editor.html:54-169` (no skip-link element)
**Pain:** No `<a class="skip-link" href="#workspace">Skip to main</a>` exists. Keyboard users entering the page from address bar have to Tab through topbar (12+ stops including overflow menu, theme, undo, redo, open, present, export, export-pptx, overflow trigger) before reaching the slide rail or preview. Restore banner adds 2 more stops when present.
**WCAG ref:** 2.4.1 Bypass Blocks (Level A).
**Evidence:** Body opens with `<div id="restoreBanner">` (line 60), then `<header class="topbar">` (line 81). First focusable inside main is at line 240+.
**Fix direction:** S — add `<a id="skipToMain" class="skip-link" href="#workspace">Перейти к рабочей области</a>` as first child of `<body>`; CSS `.skip-link { position:absolute; left:-9999px; } .skip-link:focus { left:8px; top:8px; ... }`.
**Reuse note:** Standard pattern; cleaner than introducing a new landmark.

### F7 — HIGH — WCAG 4.1.2 (A) — Floating toolbar shown without ensuring keyboard reachability after selection
**File:** `editor/src/floating-toolbar.js:202-265`; `editor/presentation-editor.html:660-842`
**Pain:** `#floatingToolbar` has `role="toolbar"` ✓ and `aria-label="Быстрые действия"` ✓. On selection: `toolbar.hidden = false; toolbar.setAttribute("aria-hidden", "false")` (line 203-204). But there is NO mechanism to move focus to it from a keyboard-driven selection (Enter on rail item activates slide, not the floating toolbar). User can only reach it by tabbing OUT of the iframe back through inspector, into preview area — depending on tab order it may be unreachable in practice without a known shortcut.
**WCAG ref:** 4.1.2 Name/Role/Value (A) + 2.1.1 Keyboard (A).
**Evidence:** No `tabindex` on `#floatingToolbar`, no documented keyboard shortcut to focus it (Shortcuts modal HTML:1867-1920 doesn't mention one).
**Fix direction:** M — wire keyboard activation: e.g. F10 / Alt+Shift+T to focus first enabled toolbar button when toolbar visible; document in shortcuts table.
**Reuse note:** Toolbar already has correct ARIA — only keyboard discoverability missing.

### F8 — HIGH — WCAG 1.4.3 (AA) — `--shell-text-muted` (light theme) measured against `--shell-field-bg`, not just `--shell-bg`
**File:** `editor/styles/tokens.css:27, 37`; `editor/styles/base.css:456-466`
**Pain:** `.mini-badge { background: var(--shell-field-bg); color: var(--shell-text-muted); font-size: 12px; }` — `--shell-field-bg = #f2f2f7` and `--shell-text-muted = rgba(60,60,67,0.78)`. The v2.0.16 fix was measured on white (#ffffff). Effective foreground on #f2f2f7 ≈ `#52525c` → contrast ≈ 4.6:1 — JUST passes AA (4.5:1). On `--shell-panel-soft` (#f9f9fb) it drops to ~5.0:1. Borderline; any future dark-mode rebind could regress.
**WCAG ref:** 1.4.3 Contrast (AA) — normal text ≥ 4.5:1.
**Evidence:** Existing `contrast.spec.js` only tests text-muted on `--shell-bg` and `--shell-panel`; never on `--shell-field-bg` (where mini-badges actually render).
**Fix direction:** S — extend `contrast.spec.js` `--shell-text` and `--shell-text-muted` against `--shell-field-bg` (light + dark). No CSS change yet — measure first.
**Reuse note:** Builds on v2.0.16 work without re-doing it.

### F9 — MED — WCAG 1.3.1 (A) — Heading hierarchy skipped inside `#openHtmlModal` (h3 → h4 with no h3 in between)
**File:** `editor/presentation-editor.html:1670, 1688, 1700, 1723`
**Pain:** Modal has `<h3>Открыть HTML</h3>` (line 1670) followed inside `.split-grid` by two `<h4>Загрузка файла</h4>` and `<h4>Вставка HTML</h4>` and `<h4>Папка ресурсов</h4>` (lines 1688, 1700, 1723). Per WCAG 1.3.1 + 2.4.6, heading levels should descend without skipping; here h4 has no parent h3 *within the same logical group* (the global h3 is the modal title). axe doesn't flag this consistently but screen-reader heading navigation skips an entire level. Same pattern in `#videoInsertModal:1817, 1831`.
**WCAG ref:** 1.3.1 Info & Relationships (A); 2.4.6 Headings & Labels (AA).
**Evidence:** `<h3>Открыть HTML</h3> ... <h4>Загрузка файла</h4>`.
**Fix direction:** S — make modal title an h2 (semantic top of dialog when role=dialog applied per F1) so h3 children align — OR change h4s to h3s consistently.
**Reuse note:** Coordinate with F1 fix (modal dialog role) — same 4 files.

### F10 — MED — WCAG 4.1.3 (AA) — Toast container uses `aria-live="polite" aria-atomic="false"` but no `role="status"` and no `role="alert"` for errors
**File:** `editor/presentation-editor.html:1958-1966`; `editor/src/feedback.js:4-53`
**Pain:** `#toastContainer` is `aria-live="polite" aria-atomic="false"`. For errors (`type === "error"`, line 51-52: 5 200 ms TTL) the toast should be announced more assertively via `role="alert"` (implicit `aria-live="assertive"` + `aria-atomic="true"`). Currently warnings and errors get the same polite delivery; users with screen readers may miss critical "Действие заблокировано" feedback.
**WCAG ref:** 4.1.3 Status Messages (AA).
**Evidence:** `<div id="toastContainer" class="toast-container" aria-live="polite" aria-atomic="false">`; in `feedback.js`, the toast div doesn't get its own role per type.
**Fix direction:** M — when `type === "error"`, add `toast.setAttribute("role", "alert")` (or render into a parallel `#toastContainerAssertive` with `aria-live="assertive"`).
**Reuse note:** Independent; the `aria-live` foundation from v2.0.22 covers polite path only.

### F11 — MED — WCAG 4.1.2 (A) — `mode-toggle` buttons missing initial `aria-pressed` in HTML (set only by JS at boot)
**File:** `editor/presentation-editor.html:124-131, 866-871, 970-981`
**Pain:** Three `mode-toggle` groups (`#previewModeBtn/#editModeBtn`, `#basicModeBtn/#advancedModeBtn`, `#smartModeBtn/#containerModeBtn`) rely on `setToggleButtonState` (`shell-layout.js:30-34`) to apply `aria-pressed`. If JS fails to load (corrupt cache, CSP block) the toggle state is invisible to AT. Pre-set `aria-pressed="true"` on the `is-active` buttons and `aria-pressed="false"` on the rest at HTML author time.
**WCAG ref:** 4.1.2 Name/Role/Value (A); progressive enhancement.
**Evidence:** `<button type="button" id="previewModeBtn" class="is-active"><span>Превью</span></button>` — `class="is-active"` but no `aria-pressed`.
**Fix direction:** S — add `aria-pressed="true"` to the 3 `is-active` buttons and `aria-pressed="false"` to the 3 sibling buttons in HTML. JS continues to flip dynamically.
**Reuse note:** Independent of v2.0.22.

### F12 — MED — WCAG 2.4.3 (A) — Roving tabindex within rail breaks Shift+Tab return path
**File:** `editor/src/slide-rail.js:43, 162-181`
**Pain:** Slide rail uses roving tabindex correctly (one item with `tabindex="0"`, others `-1`). But after the user lands on a non-active item via Arrow keys (line 177-178: `items[nextPos].setAttribute("tabindex", "0"); items[nextPos].focus()`), pressing Shift+Tab returns to the **previous** focusable in DOM order (the slide-template trigger / panel-close) rather than to the topbar. This is correct behavior for roving tabindex BUT users who navigated INTO the rail via Tab from the topbar expect Shift+Tab to be a symmetric back-out. Verify with NVDA + JAWS testing.
**WCAG ref:** 2.4.3 Focus Order (A).
**Evidence:** Lines 162-180 — the only focusable in #slidesList becomes the new active item; all other DOM-prior elements still take precedence on Shift+Tab.
**Fix direction:** L — could be a roving-tabindex misunderstanding (intended by ARIA pattern). Recommend writing a test that asserts: after ArrowDown 3x then Shift+Tab → focus returns to `#toggleSlideTemplateBarBtn` (the immediate prior focusable). If that's the intended contract, document it; if not, restore tabindex=0 to the previously-active item on focus loss.
**Reuse note:** Builds on existing P0-08 invariant test in `keyboard-nav.spec.js`.

### F13 — MED — WCAG 4.1.2 (A) — Selection frame has tabindex=0 + 8 nested resize-handle buttons (potential focus trap on Tab)
**File:** `editor/presentation-editor.html:587-657`; `editor/src/selection.js:54`
**Pain:** `#selectionFrame` is `tabindex="-1"` in HTML but JS toggles to `tabindex=0` when visible (selection.js:54). The frame contains 8 `<button class="selection-handle">` (NW, N, NE, E, SE, S, SW, W). When the frame is focused via Tab, pressing Tab again moves into the resize handles — there is no focus exit shortcut documented. ALSO: when overlay is locked (`overlayLocked`), `els.selectionHandles.forEach((handle) => { handle.disabled = ...; handle.style.pointerEvents = ... })` (line 89-93) — disabled handles are skipped by Tab automatically (good), but the frame itself remains `tabindex=0`. Combined Tab visit count: 1 frame + up to 8 handles = 9 stops per selected element. This is excessive and conflicts with the iframe's own keyboard handling.
**WCAG ref:** 4.1.2 + 2.4.3 Focus Order.
**Evidence:** Lines 587-657 — 8 sequential `<button class="selection-handle">`.
**Fix direction:** M — group 8 handles into one virtual control with `aria-keyshortcuts="ArrowKey"` on parent; OR set `tabindex="-1"` on all handles when `overlayLocked` AND on the frame when no resize policy. Current state burdens keyboard users.
**Reuse note:** Existing CSS focus-visible rules cover styling; this is a tab-stop count issue.

### F14 — MED — WCAG 4.1.2 (A) — `#selectionFrame` with `aria-hidden="true"` then `aria-hidden="false"` but element is still inside an `aria-hidden=true` parent in some states
**File:** `editor/presentation-editor.html:585-657`; `editor/src/selection.js:12, 18`
**Pain:** `#selectionOverlay` has `aria-hidden="true"` initially. JS toggles both `#selectionOverlay` and `#selectionFrame` aria-hidden at the same time. Focus while ancestor `aria-hidden=true` is invalid per ARIA spec — but `tabindex=0` makes the frame focusable. Race condition window: between `tabindex=0` assignment and aria-hidden=false, the frame may briefly be focusable-while-hidden.
**WCAG ref:** 4.1.2 + ARIA spec §10.6.
**Evidence:** Two synchronous setAttribute calls (selection.js:12, 18) inside a single `renderSelectionOverlay()` call — should be atomic, but the order matters; aria-hidden change must precede tabindex.
**Fix direction:** S — flip aria-hidden BEFORE setting tabindex; defer tabindex via `requestAnimationFrame` if needed. Defensive only.
**Reuse note:** Independent of foundation work.

### F15 — LOW — WCAG 2.4.7 (AA) — Focus ring uses `box-shadow` not `outline`, may overflow `overflow:hidden` parents
**File:** `editor/styles/base.css:174-188`
**Pain:** `button:focus-visible { outline: none; box-shadow: 0 0 0 3px ...; }`. Inside `inspector-body { overflow: auto; }` (inspector.css:9) the 3px box-shadow ring is clipped on focused buttons that are flush with the section edge. `outline` would render outside the layout box and avoid clipping; `box-shadow` is part of the box model.
**WCAG ref:** 2.4.7 Focus Visible (AA).
**Evidence:** `inspector.css:9 overflow: auto` + `base.css:187 box-shadow: 0 0 0 var(--focus-ring-width, 3px)`.
**Fix direction:** S — use `outline: var(--focus-ring-width) solid var(--focus-ring-color); outline-offset: 2px;` instead of box-shadow. Modern outline fully supports rounded corners (outline follows border-radius since Chrome 94 / Firefox 88 / Safari 16).
**Reuse note:** v2.0.22 boosted ring alpha — this is the rendering-mechanism follow-up.

### F16 — LOW — WCAG 2.4.6 (AA) — `#emptyStarterDeckBtn` aria-label duplicates visible text
**File:** `editor/presentation-editor.html:545-553`; `editor/src/onboarding.js:48`
**Pain:** Button `<button id="emptyStarterDeckBtn" class="text-link-btn" aria-label="Открыть стартовый пример">попробуйте на примере</button>`. The aria-label "Открыть стартовый пример" overrides the visible text "попробуйте на примере" — voice-control users who say "click попробуйте на примере" cannot activate it, because the accessible name comes from aria-label.
**WCAG ref:** 2.5.3 Label in Name (Level A — yes, A not AA — voice control).
**Evidence:** Line 549-552.
**Fix direction:** S — drop aria-label entirely (visible text is sufficient: "попробуйте на примере"); keep `title` if extra context is desired.
**Reuse note:** Independent.

### F17 — LOW — WCAG 1.4.11 (AA) — Disabled buttons rely solely on `opacity: 0.55` for state (no non-color cue)
**File:** `editor/styles/base.css:109-115`
**Pain:** `button:disabled { opacity: 0.55; cursor: not-allowed; }`. Color contrast at 55% opacity for `.primary-btn` (white text on accent blue) drops to ~3.0:1 (fails AA for non-text UI components). Disabled state is conveyed by opacity alone — no icon, no aria-disabled label change. Screen readers do read "disabled", so AT users are OK; sighted-but-low-vision users may miss the state.
**WCAG ref:** 1.4.11 Non-text Contrast (AA).
**Evidence:** `opacity: 0.55` + no border style change.
**Fix direction:** M — for `.primary-btn:disabled`, also set `background: var(--shell-field-muted); color: var(--text-disabled)`. Removes opacity ambiguity and improves contrast.
**Reuse note:** v2.0.16 added `--text-disabled` token; reuse it here.

### F18 — AAA — WCAG 1.4.6 — `--shell-text-muted` 5.5:1 doesn't meet AAA (7:1) on small text
**File:** `editor/styles/tokens.css:37`
**Pain:** Current 5.5:1 passes AA but not AAA. For users with low vision who'd benefit from AAA, the muted text (panel-subtext, help text, hints) is technically borderline. v2.0.16 chose AA as the explicit target — note for future hardening.
**WCAG ref:** 1.4.6 Contrast (Enhanced) — Level AAA, normal text ≥ 7:1.
**Evidence:** Token alpha 0.78 → ~5.5:1 on white.
**Fix direction:** L — would need alpha 0.92+ (~7:1) to hit AAA. Decisional, not a current bug.
**Reuse note:** Document as deliberate AA-not-AAA stance.
